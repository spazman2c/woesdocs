# Broadcasts Overview

Broadcasts are customer-facing announcements or follow-ups sent to a segment of
customers. They are useful when a product change, docs update, incident
resolution, or feature release should reach more than one customer.

<Note>
  Broadcast availability can vary by workspace while customer update surfaces
  are rolled out. Use this section to plan the workflow and keep message content
  tied to issues, accounts, and product context.
</Note>

## Good Broadcast Use Cases

- A feature request has shipped.
- A known API issue has been resolved.
- Documentation or SDK examples were updated.
- Customers need a migration reminder.
- A workaround changed or is no longer required.

## Broadcast Inputs

| Input | Why it matters |
| --- | --- |
| Audience | Which customers or accounts should receive the update. |
| Reason | The issue, feature, docs gap, or incident behind the message. |
| Source of truth | Link to docs, release notes, issue, or internal decision. |
| Required action | Whether customers need to update code, retry, or do nothing. |
| Success metric | Replies, resolved issues, lower repeat questions, or survey improvement. |
