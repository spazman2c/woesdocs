# Create a Broadcast

A good broadcast is specific, targeted, and tied to a real customer need. Avoid
generic marketing copy; customers should understand what changed and whether
they need to act.

## Message Structure

| Section | Include |
| --- | --- |
| Subject | The product area, change, or action required. |
| Summary | One or two sentences explaining what changed. |
| Impact | Who is affected and how. |
| Action | What the customer should do next, if anything. |
| Link | Updated docs, issue status, release note, or support thread. |

## Draft Checklist

- The audience is specific.
- The message has a clear reason.
- The customer impact is accurate.
- The required action is explicit.
- Links point to current docs or product pages.
- No private customer data or internal-only notes are included.
- Support operators know how to respond to replies.

## Example Outline

```text
Subject: Webhook retry behavior is now documented

We updated the webhook documentation with retry timing, delivery headers, and
example payloads for failed deliveries.

If your team handles webhook retries manually, review the updated docs before
changing retry logic. No production change is required.
```
