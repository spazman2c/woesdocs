# Automations

Automations are workspace-level workflow rules. They help route support work, apply labels, set conversation lifecycle state, request surveys, schedule follow-up, and record operator-visible notes.

Automations are managed from the primary **Automations** surface, not inside Settings.

## Permissions

| Permission | Access |
| --- | --- |
| `automations.read` | View automation rules, tests, history, delay queues, and logs. |
| `automations.write` | Create, edit, test, publish, unpublish, and delete automations. |

Owners and Admins receive both permissions by default. Members and Viewers receive read access by default.

## Automation Structure

An automation contains:

- One **When** trigger event.
- Optional **If** or **AI condition** nodes.
- Optional **Delay** nodes.
- One or more **Then** action nodes.
- Version and publish state.
- Test status and last-tested timestamp.
- History, delay queue, and run logs.

New automations start as drafts. A draft can be edited and tested, but it does not run for customers until it is published. A published automation is live when it is enabled and has a published version.

## Common Recipes

Use these as starting points, then adjust conditions for your workspace.

### Route New API Questions To AI

| Step | Node |
| --- | --- |
| 1 | When: **New customer message** or **New live chat conversation** |
| 2 | AI condition: **Customer intent is API question** |
| 3 | Then: **Hand to AI agent** |
| 4 | Then: **Add tag** such as `api-question` |

### Escalate Missing Context

| Step | Node |
| --- | --- |
| 1 | When: **New customer message** |
| 2 | AI condition: **AI outcome is handed-off** |
| 3 | Then: **Add tag** such as `needs-context` |
| 4 | Then: **Human handoff** |
| 5 | Then: **Notify** the owning team |

### Collect CSAT After Resolution

| Step | Node |
| --- | --- |
| 1 | When: **Conversation solved** |
| 2 | Delay: wait long enough for the customer to read the resolution |
| 3 | If condition: conversation does not already have the survey exclusion tag |
| 4 | Then: **Send CSAT survey** |

### Reopen When A Customer Replies

| Step | Node |
| --- | --- |
| 1 | When: **New customer message** |
| 2 | If condition: **Status is solved** |
| 3 | Then: **Set conversation status** to `open` |
| 4 | Then: **Assign team** or **Notify** |

### Move To Waiting After An Operator Reply

Use this when your support workflow treats an operator reply as "waiting on the customer."

| Step | Node |
| --- | --- |
| 1 | When: **Operator replied** |
| 2 | Optional If condition: **Channel is Live chat**, **Priority is urgent**, or **Tag contains vip** |
| 3 | Then: **Set conversation status** to `waiting` |
| 4 | Optional Then: **Add internal note** such as "Waiting on customer after operator response." |

### Snooze After An Operator Reply

Use this when you want the reply to temporarily move the conversation out of the active queue.

| Step | Node |
| --- | --- |
| 1 | When: **Operator replied** |
| 2 | Then: **Snooze conversation** |
| 3 | Choose the snooze duration: `1h`, `4h`, `1d`, `3d`, or `1w` |

### Escalate SLA Breaches

| Step | Node |
| --- | --- |
| 1 | When: **SLA breached** or **No first response in time** |
| 2 | If condition: **Priority is urgent** or **Channel is live chat** |
| 3 | Then: **Notify** |
| 4 | Then: **Assign team** |
| 5 | Then: **Add internal note** |

## Assignment And Routing

Automations can hand work to the AI agent, hand it back to a human operator, or route it to a team. Use [Assignment](/settings/assignment) when you need persistent routing rules with skills, capacity, schedules, or route previews.

Automations should be used for event-driven workflow changes: tags, notes, notifications, surveys, SLA metadata, conversation status, priority, team routing, AI handoff, human handoff, snooze, and archive.

## Testing Automations

Use the built-in test flow before publishing. Woes creates a sandbox conversation, builds an event that satisfies the automation filters, applies the workflow, verifies the expected effects, and removes the sandbox.

<Warning>
  Automation tests are designed to avoid production data mutation. Do not test by manually triggering rules on real customer records unless you intend the workflow to run.
</Warning>

## Logs And History

Automations keep history and run evidence so operators can understand what happened:

| Surface | What it shows |
| --- | --- |
| Last test | Latest draft test status and date. |
| History | Publish and edit history for the automation. |
| Logs | Runtime automation runs collected from conversation metadata. |
| Delay queue | Delayed work waiting for the scheduled automation processor. |

Logs are best-effort operational evidence. Automations should never block the customer or operator path if a workflow fails to apply.
