# Automations

Automations are workspace-level workflow rules. They help route support work, apply labels, set lifecycle state, and trigger follow-up actions.

Automations are managed from the primary **Automations** surface, not inside Settings.

## Permissions

| Permission | Access |
| --- | --- |
| `automations.read` | View automation rules and logs. |
| `automations.write` | Create, edit, publish, unpublish, and delete automations. |

Owners and Admins receive both permissions by default. Members and Viewers receive read access by default.

## Automation Structure

An automation contains:

- Kickoff event.
- Optional conditions.
- One or more ordered paths.
- Optional delays.
- Actions.
- Version and publish state.
- Logs and test results.

## Supported Node Catalog

The node library contains configurable node types, not duplicate preset nodes.
For example, use **Set priority** and choose a priority value instead of adding
separate "Set high priority" or "Set low priority" nodes. Use **Snooze issue**
and choose a duration instead of separate one-hour or four-hour snooze nodes.

| Node | Type | Use it for |
| --- | --- | --- |
| Trigger | Start | Start from a supported runtime event. |
| If condition | Logic | Branch on conversation, issue, customer, message, survey, or tag fields. |
| AI condition | Logic | Branch on detected topic or AI outcome. |
| Delay | Timing | Wait minutes, hours, or days before continuing. |
| Hand to AI agent | Routing | Let the Woes AI assistant handle the conversation. |
| Human handoff | Routing | Stop AI ownership and leave the conversation for an operator. |
| Assign team | Routing | Route work to a team queue. |
| Set status | Ticket action | Move lifecycle state to a configured value. |
| Set priority | Ticket action | Set the configured priority value. |
| Add tag | Ticket action | Apply one normalized issue tag. |
| Auto-reply | Message action | Add a configured assistant reply when the AI has not already answered. |
| Notify | Operator action | Record an operator notification in automation notes. |
| Send CSAT survey | Survey action | Request an eligible CSAT survey. |
| Send NPS survey | Survey action | Request an eligible NPS survey. |
| Set SLA target | SLA action | Apply first-response or resolution timing metadata. |
| Add internal note | Operator action | Write an operator-only note. |
| Snooze issue | Lifecycle action | Snooze until the configured future time. |
| Archive issue | Lifecycle action | Move the conversation out of the active inbox. |

## Trigger Nodes

Trigger nodes are the only starting point for an automation. Woes only exposes trigger events that are fired by the runtime.

| Trigger | When it runs |
| --- | --- |
| Issue created | A new issue or support conversation is created. |
| New live chat conversation | A visitor starts a new live chat conversation. |
| Customer replied | A customer sends a new message on an existing issue or conversation. |
| New Discord message | A Discord message is routed into the Woes inbox. |
| Issue status changed | Status changes to open, waiting, or solved. |
| Issue solved / reopened / snoozed / archived | The matching lifecycle action happens in the inbox. |
| Priority changed / issue assigned / tag added | The matching operator-side field changes. |
| SLA breached / no first response | The automation scheduler detects the SLA event. |
| CSAT completed / NPS completed | A customer submits the matching survey response. |

## Logic Nodes

| Node | Use it for |
| --- | --- |
| If condition | Branch on fields from the issue or conversation. |
| AI condition | Branch on detected customer intent or AI outcome. |
| Delay | Wait minutes, hours, or days before continuing. Support-hours delay basis is stored with the automation, but current runtime processing treats the delay as elapsed time. |

Supported condition fields are priority, status, channel, assignee, tag, customer email, message text, message mentions workspace, survey type, survey score, CSAT score, AI topic, and AI outcome.

AI topic values are `api-question`, `human-request`, `off-topic`, `greeting`, `closing`, and `abusive`. AI outcome values are `answered`, `handed-off`, and `resolved`.

## Action Nodes

| Action | Result | Required configuration |
| --- | --- | --- |
| Hand to AI agent | Assigns the conversation to the Woes AI assistant. | None. |
| Human handoff | Stops AI ownership and leaves the issue for an operator. | None. |
| Assign team | Routes the issue to a team. | Team name. |
| Set status | Moves lifecycle state. | Status value. |
| Set priority | Updates priority. | Priority value. |
| Add tag | Applies a normalized issue tag. | Tag value. |
| Auto-reply | Inserts a delivered assistant message when the AI has not already answered. | Reply body. |
| Notify | Records an operator notification in automation notes. | Recipient target; message is optional. |
| Send CSAT survey | Requests customer satisfaction feedback. | Optional automation survey selection. |
| Send NPS survey | Requests recommendation feedback. | Optional automation survey selection. |
| Set SLA target | Applies first-response or resolution timing metadata. | SLA target. |
| Add internal note | Writes an operator-only note. | Note body. |
| Snooze issue | Moves the item into waiting until a future time. | Snooze duration. |
| Archive issue | Moves the conversation out of the active inbox. | None. |

<Note>
  Survey actions mark the conversation as requesting an eligible survey. The
  widget and survey settings still decide whether a survey can be shown for that
  customer and channel.
</Note>

## Common Recipes

Use these as starting points, then adjust conditions for your workspace.

### Route New API Questions To AI

| Step | Node |
| --- | --- |
| 1 | Trigger: **Issue created** or **New live chat conversation** |
| 2 | If condition: `channel` is `live chat` and `message text` mentions an API topic |
| 3 | Action: **Hand to AI agent** |
| 4 | Action: **Set priority** to `Medium` |

### Escalate Missing Context

| Step | Node |
| --- | --- |
| 1 | Trigger: **Customer replied** |
| 2 | AI condition: `AI outcome` is `handed-off` |
| 3 | Action: **Add tag** such as `needs-context` |
| 4 | Action: **Human handoff** |
| 5 | Action: **Notify** the owning team |

### Collect CSAT After Resolution

| Step | Node |
| --- | --- |
| 1 | Trigger: **Issue solved** |
| 2 | Delay: wait until the customer has had time to read the resolution |
| 3 | If condition: issue does not already have the survey exclusion tag |
| 4 | Action: **Send CSAT survey** |

### Reopen When A Customer Replies

| Step | Node |
| --- | --- |
| 1 | Trigger: **Customer replied** |
| 2 | If condition: `status` is `solved` |
| 3 | Action: **Set status** to `Open` |
| 4 | Action: **Assign team** or **Notify** |

## Assignment And Routing

Automations can hand work to the AI agent, hand it back to a human operator, or
route it to a team. Use [Assignment](/settings/assignment) when you need
persistent routing rules with skills, capacity, schedules, or route previews.

Automations do not currently assign directly to an account owner or arbitrary
workspace user. Keep those decisions in team assignment rules or manual triage.

## Testing Automations

Use the built-in test flow before publishing. Woes creates a sandbox record, applies the automation, verifies each expected effect, then removes the sandbox.

<Warning>
  Automation tests are designed to avoid production data mutation. Do not test by manually triggering rules on real customer records unless you intend the workflow to run.
</Warning>
