# Users and Permissions

Users and Permissions help admins control who can read, reply, configure, and manage the workspace.

## Users

The Users page shows:

- Member identity.
- Role.
- Whether access is inherited from the role or customized for that user.
- Status.
- Current viewing target where available.
- Links to jump to the item a user is viewing.

## Permissions

Woes uses section-specific permissions. This keeps broad settings access from becoming a single all-or-nothing control.

| Permission area | Examples |
| --- | --- |
| Inbox | Read conversations, reply, take over, manage customer interactions. |
| Automations | View or write automation rules. |
| Surveys | Configure CSAT and NPS surveys. |
| Widget | Configure widget install and behavior. |
| Keys | Manage API keys and widget key surfaces. |
| Billing | Open subscription management and change paid capacity through workspace write access. |
| Data | Tags, statuses, custom fields, and priority labels. |
| Macros | Create, update, or delete macros. |
| Team Chat | Read internal chat, send messages, and manage channels where allowed. |
| Security | Read logs and configure security/redaction surfaces where allowed. |

## Roles And Per-User Access

Most users should inherit permissions from a role. Roles are easier to audit and
make access predictable across the workspace.

Admins can also customize access for a specific user when one person needs a
temporary or exceptional permission set. Per-user access overrides the selected
role's permissions for that member until the override is removed.

Use per-user access for cases like:

- Temporarily giving an operator billing or keys access.
- Letting a support lead configure surveys without changing every Admin.
- Removing a sensitive write permission from one user while keeping their role.
- Testing a new permission profile before creating a new role.

<Warning>
  Per-user access is powerful. Prefer role defaults for normal team structure
  and reserve direct overrides for exceptions that are reviewed regularly.
</Warning>

### Customize A User

<Steps>
  <Step title="Open Users">
    Go to **Settings → Users**.
  </Step>
  <Step title="Open the user menu">
    Choose **Customize user access** or **Edit user access** from the row menu.
  </Step>
  <Step title="Copy role access">
    If the user inherits role permissions, copy the role access first to start
    from the current role defaults.
  </Step>
  <Step title="Adjust permissions">
    Expand permission categories, toggle read/write/execute/delete permissions,
    or bulk enable and disable a category.
  </Step>
  <Step title="Return to role defaults">
    Use **Use role defaults** when the user should inherit the role again.
  </Step>
</Steps>

Changing per-user access requires permission to manage both users and roles.
The workspace owner profile cannot be stripped through per-user overrides.

## Current Viewing State

Woes can show what a user is currently viewing across conversations, issues, and supported platform areas. This helps admins coordinate live work and jump directly to the relevant item.

Viewing state updates in real time and naturally expires when a tab stops
heartbeat updates. The visible state can include:

- Conversation pages.
- Issue detail views.
- Supported workspace pages.

When a user is viewing a conversation or issue, admins can click the target from
the Users table to open the same item.

## Recommended Access

- Give Owners full setup and permission access.
- Give Admins channel, agent, billing, automation, and workspace configuration access.
- Give Members inbox and issue workflow access.
- Give Viewers read-mostly access for audits or leadership visibility. The default Viewer role can still send Team Chat messages.
- Review direct user overrides during security and billing reviews.

## Runtime Gating

The access matrix controls actual product behavior, not just navigation.
For example, Inbox read access lets a user inspect conversations, while
`inbox.reply`, `inbox.archive`, `inbox.delete`, and `customers.write` control
the composer, takeover, archive/delete actions, tags, notes, and customer-field
edits. Users without a required action permission see a read-only or disabled
state instead of a working control.

Team Chat uses `chat.read` to open internal conversations, `chat.send` to send
messages, reactions, attachments, and read receipts, and
`chat.channels.manage` to create or administer channels where membership rules
allow it.

Security uses `security.read` for Settings → Security and Logs and history.
`security.write` protects security and redaction configuration changes.
