# Sentry Connector

Sentry is a planned customer connector for connecting developer support conversations to application errors and production issue evidence.

## Status

**Planned connector.** This is not currently a live Woes connector. It should remain disabled until project access, event lookup, permissions, redaction, and audit behavior are implemented.

## What it will do

A Sentry connector should help operators answer error-driven support questions faster:

- Attach relevant Sentry issues or events to Woes conversations.
- Search recent errors by customer, release, environment, route, or error fingerprint where safe.
- Add operator-visible error context to support issues.
- Trigger automations when a customer reports a known production error.
- Link back to the Sentry issue for engineering investigation.

## Product fit

Many API support questions begin with an error message, failed webhook, or unexpected server response. Connecting Woes to error telemetry can help distinguish documentation confusion from a real product failure.

## Implementation requirements

Before Sentry becomes active, Woes needs:

- Project and environment selection.
- Workspace-scoped OAuth or token installation.
- Customer-safe matching rules for users, accounts, releases, and routes.
- Redaction of stack frames, request data, and metadata that should not leave operator-only context.
- Rate limits and payload bounds for issue and event lookup.
- Audit logs for attach, detach, and automation-driven actions.

## Possible first version

Start with read-only issue lookup and manual attachment from the operator panel. Keep automatic event ingestion and alert-triggered issue creation for a later version.
