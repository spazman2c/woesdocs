# API Context Sources

API context sources are the materials Woes scans so agents and operators can answer developer questions with grounded evidence.

## Status

**Available now.** Woes can ingest multiple source types through the Context Library and then attach those sources to agents.

## Supported source families

| Source family | Use it for |
| --- | --- |
| OpenAPI or Swagger | Endpoint paths, methods, parameters, request bodies, schemas, examples, auth hints, and responses. |
| Postman | Collections and examples that describe real integration workflows. |
| GraphQL | Schema-driven operations, types, fields, and resolver documentation. |
| GitHub | Repository docs, SDK examples, changelogs, and source-adjacent usage notes. |
| Website or docs pages | Public API documentation, guides, and troubleshooting pages. |
| Manual or pasted content | Hand-curated support notes, onboarding rules, and product-specific guidance. |

## Where it appears in Woes

- **Context Library** creates, scans, rescans, searches, and inspects sources.
- **Settings -\> Integrations** shows source-family status for OpenAPI, Postman, GraphQL, and GitHub.
- **Agents** attaches sources to the specific agent that should use them.
- **Analytics -\> Sources** shows readiness and coverage signals.

## How Woes uses source context

When a customer asks an API question, Woes retrieves workspace-scoped source chunks and endpoint evidence. The agent should answer only when the evidence is strong enough; otherwise it asks a focused clarification or hands off.

## Operational notes

- Keep credentials separate from source content.
- Use source auth settings for guarded test calls instead of embedding secrets in docs.
- Rescan sources after major API releases or docs changes.
- Attach only relevant sources to each agent so routing stays precise.
