# Integrations Catalog

Woes integrations connect the support workspace to the systems where developer-support context and workflows already live: customer channels, repository documentation, API source files, team handoffs, billing, public API integrations, and AI-tool access.

This catalog is intentionally explicit about status. A connector should only look actionable when the Woes backend can support the action safely for a workspace.

## Status labels

| Status | Meaning |
| --- | --- |
| Available now | The integration has a working Woes surface or supported setup path today. |
| Planned connector | The integration appears in the product catalog but remains disabled until OAuth, install state, routes, permissions, audit trails, delivery behavior, and failure states are implemented. |
| Future candidate | A researched integration direction that fits the platform, but is not yet a product commitment or active connector. |

## Integration principles

- Channel integrations should create or continue records in the same Woes conversation and message model.
- Context integrations should add workspace-scoped evidence for the support agent without exposing credentials or private data to customers.
- Developer integrations should use server-side `woesk_` API keys, never widget public keys or browser code.
- Workflow integrations should be disabled until actions, permissions, audit trails, and failure states are fully implemented.
- Customer-facing docs should never claim an integration is live before the product can actually connect it.

## Where to manage integrations

Use **Workspace Settings -\> Integrations** for the product catalog and connector status. Use **Context Library** for API context imports and source management. Use the channel settings pages for live chat, email, and Discord behavior. Use **Settings -\> Keys** for widget and workspace API keys, and **Settings -\> Billing** for Stripe-backed billing state.
