# Integrations Catalog

Woes integrations are customer-managed connectors to external systems that add support context, trigger workflow actions, notify teams, or attach operational evidence to a Woes workspace.

This catalog is intentionally limited to connectors a customer can install, authorize, subscribe to, or evaluate as a future customer-facing connector. Internal platform services do not belong here. Billing systems, email delivery infrastructure, hosting, observability used by Woes itself, and provider-specific backend plumbing should stay in Billing, Channels, Security, or developer reference docs instead of the Integrations tab.

A connector should only look actionable when the Woes backend can support the action safely for a workspace.

## Status labels

| Status | Meaning |
| --- | --- |
| Available now | The connector has a working Woes surface or supported setup path today. |
| Planned connector | The connector belongs in the customer catalog, but remains disabled until OAuth or installation, workspace mapping, permissions, routes, audit trails, and failure states are implemented. |
| Additional connector candidate | A researched customer-facing connector direction that fits Woes, but is not yet a committed product surface. |

## Available now

| Connector | What it does today |
| --- | --- |
| [GitHub](./github) | Imports repository docs, examples, SDK material, and source-adjacent notes into Woes API context. |
| [Discord](./discord) | Connects selected Discord support activity to the shared Woes inbox and routing model. |

## Connector principles

- Customer connectors should be workspace-scoped and safe to install, disconnect, audit, and troubleshoot from Woes.
- Context connectors should add evidence for operators and agents without exposing credentials or private data to customers.
- Workflow connectors should stay disabled until actions, permissions, retries, audit trails, and failure states are fully implemented.
- Event connectors should use signed payloads, bounded schemas, retry controls, and redacted delivery logs.
- Customer-facing docs should never claim a connector is live before the product can actually connect it.

## Where to manage connectors

Use **Workspace Settings -> Integrations** for connector status and connector-level actions. Use **Context Library** for GitHub repository imports and source management. Use channel settings pages for live chat, email, and Discord behavior after a message enters Woes.
