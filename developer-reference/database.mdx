# Database

IQ Connect stores integration state inside each tenant's IQ Reseller Azure SQL database. These are tables, not separate databases. For example, `iqconnect_so` is a table in the PARTpoint tenant database.

## Core Tables

| Table | Purpose |
| --- | --- |
| `iqconnect_feedhistory` | Feed content and transmission history. Catalog content is purged sooner than order content. |
| `iqconnect_lastsync` | Last sync timestamp, record counts, and initial-load flags by feed. |
| `iqconnect_so` | Marketplace order mapping, IQ SO ID, external PO number, workflow status, feed GUID, and ETA state. |
| `iqconnect_sodetail` | SO detail mapping between IQ line numbers and external marketplace line numbers. |
| `iqconnect_posted_voupdates` | Best Buy vendor order update statuses already posted. |
| `iqconnect_batch_soupdate_resend` | Manual queue for Best Buy SO update / promise-date resend recovery. |
| `iqconnect_inventory` | Inventory availability staging for incremental sync. |
| `iqconnect_item2` | Pricing staging table. |
| `iqconnect_sc_*` | SellerCloud staging tables for inventory, items, and PO line items. |
| `integration_settings` | Per-tenant integration configuration stored as XML or JSON. |

## CRM Tables

| Table | Purpose |
| --- | --- |
| `iqconnect_crm_company` | IQ CV to CRM company mapping. |
| `iqconnect_crm_contact` | IQ contact to CRM contact mapping. |
| `iqconnect_crm_sp` | IQ SP to CRM deal mapping and status tracking. |
| `iqconnect_crm_duplicate_contacts` | Contact deduplication exclusion list. |
| `iqconnect_syncflags` | Flags used to prevent repeat processing for selected workflows. |

## Troubleshooting Pattern

When a feed appears to have skipped a record, check three things:

1. The source IQ table (`st`, `stdetail`, `inventory`, `item`, or CRM source table).
2. The relevant `iqconnect_*` tracking table.
3. The feed history or generated file output.
