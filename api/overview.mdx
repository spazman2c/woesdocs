## Base URL

```txt
https://woes.dev/api/v1
```

## Authentication

Send your workspace API key as a Bearer token.

```bash
curl https://woes.dev/api/v1/conversations \
  -H "Authorization: Bearer $WOES_API_KEY"
```

Create keys in **Settings → Keys**. Full secrets are shown once and then stored hashed.

## Test Calls In The Docs

Open any page in the **Endpoint Reference** group to test an API request from
the docs. Each endpoint page has an interactive playground with example
parameters and request bodies already filled in.

1. Paste your `woesk_` key into the Bearer token field.
2. Review the prefilled parameters or JSON body.
3. Click **Send** to run the request against `https://woes.dev`.

<Warning>
  Only use your own workspace API key. The docs use placeholder values and never
  include a real secret.
</Warning>

## Scopes

Workspace API keys can be unrestricted or scoped. Scoped keys use resource/action pairs:

| Scope                 | Allows                                    |
| --------------------- | ----------------------------------------- |
| `agents:read`         | List and read public agent metadata.      |
| `clients:read`        | List and read client account aggregates.  |
| `conversations:read`  | List and read conversations and messages. |
| `conversations:write` | Create/update conversations and messages. |
| `issues:read`         | List and read issues.                     |
| `issues:write`        | Create and update issues.                 |
| `agents:*`            | All agent operations.                     |
| `clients:*`           | All client operations.                    |
| `conversations:*`     | All conversation and message operations.  |
| `issues:*`            | All issue operations.                     |
| `*`                   | All REST API operations.                  |

<Warning>
  REST API keys are server-side secrets. Do not place `woesk_` keys in browser
  code, widget snippets, mobile apps, or customer-visible pages.
</Warning>
